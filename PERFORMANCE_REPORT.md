## 2. Consistency (10 cases √ó 3 runs)

Notebook ¬ß10 (cell 29) runs `CONSISTENCY_CASES = (SEED_TESTS + VARIANTS[:5])[:10]` three times each and reports per-axis standard deviation across the three judge verdicts. Variance is the *honest* metric ‚Äî a low mean with high variance is worse than a slightly lower mean with low variance, because production deployments need predictability.

What we look for:
- **œÉ < 0.5** on `calculation_accuracy` ‚Äî numeric answers should not move between runs.
- **œÉ < 0.8** on `instruction_adherence` and `hallucination_check` ‚Äî these are structural.
- **œÉ ‚â§ 1.2** on `scenario_completeness` and `risk_disclosure` ‚Äî these depend on LLM phrasing.

Cases that breach these bands are listed by `task_id` in the cell output and form the input for the red-team review (¬ß5).

## 3. FinOps ‚Äî Cost-per-Success vs Cost-per-Failure

Burn buckets are computed in ¬ß15 cell 35 by partitioning `_eval_results` on `stopped_reason`:

| Bucket | Definition | Typical cost/task |
|---|---|---|
| **success** | `stopped_reason="end_turn"` AND `<answer>` block present | Planner (Sonnet) + ~4 Executor turns (Haiku) + Judge (Sonnet) |
| **truncated** | `stopped_reason ‚àà {max_iterations, token_budget, wall_clock}` | Planner + ~7 Executor turns burned before guardrail trip |
| **clarify** | `stopped_reason="clarify"` (Planner routed to clarify) | Planner call only ‚Äî cheapest happy-path |
| **out_of_scope** | `stopped_reason="out_of_scope"` | Planner call only |

The exact dollar values are printed in cell 35 output and persisted to `artifacts/<ts>/burn_buckets.json`. The two operational implications:

1. **Clarify is a feature, not a fallback.** A clarified task costs ‚âà10% of a full execute task. Better to ask one question than to simulate the wrong scenario.
2. **Truncation costs more than success.** A wall-clock or iteration trip burns Planner + N Executor turns and produces nothing usable. This is why the guardrails are tight (7 iters / 60 s) rather than generous.

### High-perf vs Budget split (ADR-1 evidence)

| Role | Model | Per-task share | Why |
|---|---|---|---|
| Planner | Sonnet 4.6 | 1 call/task | Reasoning quality cascades into every downstream tool call |
| Executor | Haiku 4.5 | 3‚Äì7 calls/task | High-volume tool dispatch; format is enforced by prompt + classifier |
| Judge | Sonnet 4.6 | 1 call/task | Sloppy grading invalidates the whole evaluation matrix |

A Sonnet-everywhere variant was prototyped during development; the cost roughly tripled with no measurable lift in `calculation_accuracy` or `risk_disclosure` (the two axes that matter most). The split is the cheaper Pareto frontier.

## 4. Latency Profile (where time goes)

`CostTracker` records `inference_seconds` and `tool_seconds` separately. The breakdown is reported in cell 26 under `by_model.*.seconds` plus `tool_seconds`.

Typical profile for a successful execute task:

| Bucket | Wall fraction |
|---|---|
| Planner inference (Sonnet) | ~10‚Äì15% |
| Executor inference (Haiku, 4 turns) | ~70‚Äì80% |
| Tool execution (SQLite + pandas) | <3% |
| Judge inference (post-hoc, not on critical path) | not counted |

**Bottleneck.** Executor inference. The tool calls themselves are negligible (SQLite over a 4-table in-memory DB plus a pandas rolling mean). Latency reduction would come from cutting Executor turns, not faster tools ‚Äî which is why `simulate_scenario` returns a *complete* baseline-vs-delta-vs-verdict struct rather than three separate tools the agent would have to chain.

## 5. Red-Team Findings & Mitigations

| Attack / edge case | Original behavior | Mitigation |
|---|---|---|
| **Out-of-range date** ("Q3 2023?") | Planner v1 called `query_financials` with a 2023 filter, got zero rows, then fabricated a guess. | Planner v3: priority-ordered decision rule routes to `out_of_scope` *before* generating a plan. |
| **Binary pressure** ("Just yes or no") | Executor v2 capitulated to a one-word answer with no numbers. | Executor v3 *Conservative Bias* clause + required `<answer>` block with Numbers section. |
| **Hallucinated numbers** | Executor v1/v2 stated runway figures not backed by any tool call. | Executor v3: "if you state a number that isn't in a tool result, you have failed." Judge v3 enforces via `hallucination_check` with required trace citation. |
| **Wrong category casing** ("Marketing" vs "marketing") | Tool returned a generic error; executor gave up. | Tool now returns `{available: [...]}`; executor self-corrects on next iteration. |
| **Missing parameter** ("Should I hire someone?" with no salary) | Executor invented `$120k`. | Planner v3 routes to `clarify`. Executor never sees the underspecified plan. |
| **Infinite loop** (deliberately ambiguous prompts) | Earlier iterations hit 30+ tool calls. | `max_iterations=7` + `max_total_tokens=60_000` + `max_wall_seconds=60` ‚Äî three independent kill switches. Trips logged + counted as failures. |

### Known unfixed weakness

When the agent compares >3 scenarios in a single answer, it occasionally drops the explicit safety-threshold flag for one of them. We log this as `scenario_completeness < 5` in the Judge output rather than mask it. Future fix: structured-output mode forcing one JSON row per scenario, then format into prose.

## 6. Domain KPIs (the 3 we track)

1. **Calculation accuracy** ‚Äî runway figure within ¬±5% of synthetic ground truth.
2. **Scenario completeness** ‚Äî every scenario named in `scenarios_to_compare` has a runway figure in the `<answer> Numbers` block.
3. **Risk disclosure** ‚Äî `<answer> Risks` block names at least one of: AR variance, customer concentration, one-time vs recurring, seasonality, runway-below-threshold.

These three (not generic helpfulness scores) are what determine whether a CFO can act on the answer.
